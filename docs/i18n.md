# Internationalization (i18n)

The game is fully localized. The user picks a language on the first screen,
every user-facing string is resolved through a single locale component, and
new languages can be added by dropping in one file ‚Äî no existing code changes.

## Supported locales

| Locale | Bundle file |
|--------|-------------|
| English (`en`) | [`src/main/resources/messages_en.properties`](../src/main/resources/messages_en.properties) |
| Spanish (`es`) | [`src/main/resources/messages_es.properties`](../src/main/resources/messages_es.properties) |
| Mandarin Chinese (`zh`) | [`src/main/resources/messages_zh.properties`](../src/main/resources/messages_zh.properties) |

## How language selection works at startup

`ui.Main` boots the Swing UI in this order:

1. `LocaleManager.getInstance()` scans the classpath for every
   `messages_<lang>.properties` bundle and loads them all.
2. `LocaleSelectionFrame` opens as the first screen and shows one button per
   discovered locale, each labeled in its own language
   (`locale.getDisplayName(locale)` ‚Äî e.g. *espa√±ol*, *‰∏≠Êñá*).
3. The chosen locale becomes the active one
   (`LocaleManager.setActiveLocale`), and the player-setup screen opens with
   every string resolved via `localeManager.get(key)`.

The selection screen never hardcodes a language list ‚Äî it renders whatever
`LocaleManager.getAvailableLocales()` discovered, so it automatically shows
any new bundle.

## How strings are resolved

`domain.locale.LocaleManager` is the single source of localized text:

- `get(String key)` returns the string for the active locale and throws
  `MissingResourceException` for unknown keys, so missing translations fail
  fast instead of silently showing key names.
- `get(String key, Object... args)` applies `java.text.MessageFormat`
  placeholders, e.g. `setup.player.name.prompt=Player {0} name:`.
- UI classes (`LocaleSelectionFrame`, `PlayerSetupFrame`, `BoardFrame`) hold
  no translated text of their own; they only look up keys.

## Adding a new locale ‚Äî no code changes

The bundle filename is the contract: `messages_<lang>.properties`, where
`<lang>` is an ISO 639 language code, optionally with a region
(`messages_pt_BR.properties`).

To add French:

1. Copy `src/main/resources/messages_en.properties` to
   `src/main/resources/messages_fr.properties`.
2. Translate the values (keep the keys and `{0}`-style placeholders intact).
3. Rebuild. That's it.

`LocaleManager` discovers bundles by scanning the classpath at startup
(`discoverDefaultBundleDir` + the `messages_([a-z]{2,3}(_[A-Z]{2})?)` filename
pattern), so the new language appears on the selection screen and works
end-to-end without touching `LocaleManager`, any UI class, or the build
script. This satisfies the Open-Closed Principle: the system is open to new
locales but closed to modification.
