# Internationalization (i18n)

Players pick a language on the first screen. All UI text goes through
`LocaleManager`; adding a language is just a new properties file.

## Supported locales

| Locale | Bundle file |
|--------|-------------|
| English (`en`) | [`messages_en.properties`](../src/main/resources/messages_en.properties) |
| Spanish (`es`) | [`messages_es.properties`](../src/main/resources/messages_es.properties) |
| Mandarin (`zh`) | [`messages_zh.properties`](../src/main/resources/messages_zh.properties) |

## Startup flow

1. `Main` calls `LocaleManager.getInstance()`, which loads every
   `messages_<lang>.properties` on the classpath.
2. `LocaleSelectionFrame` shows one button per locale (label =
   `locale.getDisplayName(locale)`).
3. After the player picks one, `setActiveLocale` runs and setup/board screens
   use `localeManager.get(key)` for all strings.

We never hardcode the language list in UI code — new bundles show up
automatically.

## Lookup API

- `get(key)` — string for the active locale; missing keys throw
  `MissingResourceException`.
- `get(key, args...)` — same, with `{0}` placeholders via `MessageFormat`.
- Swing frames only store keys, not translated text.

## Adding a locale

1. Copy `messages_en.properties` → `messages_fr.properties` (example).
2. Translate values; keep keys and `{0}` placeholders.
3. Rebuild — no Java changes.

`LocaleManager` scans for `messages_*.properties` at startup, so French (or
anything else) appears on the selection screen without editing existing classes.
